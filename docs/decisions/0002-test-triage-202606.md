# ADR 0002 — Test Triage 2026-06

## Status

Accepted, 2026-06

## Context

The Ratatoskr test suite has grown to 696 test files and approximately 134K lines of test code. An ultracode triage audit was conducted in June 2026 to identify categories of tests that impose maintenance cost without providing meaningful regression coverage: characterization tests that duplicate unit-test coverage, zero-signal tests that make no behavioral assertions, provider-name-only tests that assert only on an identifier string, and written-once integration files that have never been extended beyond their initial scaffolding. This ADR records the findings and the action plan agreed upon as a result.

## Findings

### Flaky tests

None found. The codebase contains no `@pytest.mark.flaky`, `@pytest.mark.rerunfailures`, or `pytest-rerunfailures` markers. No tests were flagged as non-deterministic during the audit.

### Characterization tests

Five characterization tests were audited. All five are confirmed duplicates of existing unit or integration tests and are recommended for deletion.

| File | Test | Recommendation | Reason |
|---|---|---|---|
| `tests/characterization/test_auth_sync_characterization.py` | `test_characterization_telegram_login_response_shape_is_stable` | DELETE | `tests/api/test_auth_token_delivery.py::test_telegram_login_token_delivery` calls the same `telegram_login()` function under the same mock pattern and asserts on `data.tokens.accessToken` and `data.tokens.refreshToken` for both web and mobile clients. `tests/api/test_auth_new.py::test_telegram_login_does_not_auto_grant_owner` additionally asserts the tokens dict shape against a real DB. The characterization's extra assertions on `tokenType='Bearer'` and `sessionId` are trivially derived from the `TokenPair` model and `success_response` wrapper, both of which are exercised across dozens of other tests. |
| `tests/characterization/test_auth_sync_characterization.py` | `test_characterization_sync_session_response_shape_is_stable` | DELETE | The test injects a fake Pydantic model (`FakeSyncSession`) instead of the real `SyncService`, so it only tests the `success_response()` wrapper, not actual sync logic. `tests/api/test_sync_v2_contract.py` calls `create_sync_session()` repeatedly against a real `SyncService` and asserts `session_result["data"]["session_id"]` and `page["data"]["pagination"]["limit"]`, covering the same envelope shape with real behavior. The characterization adds no unique contract guarantee. |
| `tests/characterization/test_immediate_backlog.py` | `test_characterization_summary_command_happy_path_preserved` | DELETE | `BotSpy` and `FakeMessage` are defined in `tests/test_commands.py`, which is the canonical test for `/summarize` command routing. That file drives `bot._on_message()` with `/summarize` URLs against a real DB using the same `BotSpy` stub pattern. The characterization test covers no additional behavior — it only adds a `RuntimeConfig` fixture with `url_worker_enqueue_enabled=False`, but `test_commands.py` already covers the inline dispatch path via the same `BotSpy` mechanism. |
| `tests/characterization/test_immediate_backlog.py` | `test_characterization_youtube_uses_vtt_fallback_when_api_transcript_empty` | DELETE | `tests/test_youtube_platform_extractor.py::test_pipeline_uses_vtt_fallback_when_transcript_api_empty` tests the exact same VTT fallback scenario at the `YouTubeDownloadPipeline` level with identical mock stubs (`_extract_transcript_api` returns `('', '', False, 'api')`, `_download_video_sync` returns `subtitle_file_path`, `_load_transcript_from_vtt` returns `'vtt transcript body'`) and identical assertions (`content_source=='vtt'`, `detected_lang=='en'`, `'vtt transcript body' in content_text`). The characterization test exercises the same code path through the higher-level `YouTubePlatformExtractor.extract()` wrapper but is more fragile due to bypassing `_session_service.prepare()` via deep internal mock wiring. |
| `tests/characterization/test_immediate_backlog.py` | `test_characterization_unread_read_unread_transition_with_topic_filter` | DELETE | All three use cases are individually covered with full happy-path and error scenarios: `tests/application/use_cases/test_get_unread_summaries.py` covers topic filtering (`test_get_unread_summaries_with_topic`); `tests/application/use_cases/test_mark_summary_as_read.py` covers the read transition (`test_mark_as_read_happy_path`) and the already-read guard; `tests/application/use_cases/test_mark_summary_as_unread.py` covers the unread transition (`test_mark_as_unread_happy_path`) and the already-unread guard. The characterization's combined-cycle aspect does not expose any behavior not reachable through the individual tests. |

### Zero-signal tests

The audit identified four distinct zero-signal categories across the test suite.

| Category | Count |
|---|---|
| True zero-signal (no assertions of any kind) | 30 |
| `pytest.raises`-only (exception type asserted, no message or state check) | 332 |
| `provider_name_only` (sole assertion is on a provider identifier string) | 20 |
| Total zero-assert (all forms combined) | 684 |
| Total one-assert-no-mock | 1789 |

True zero-signal tests (count: 30) are tests that execute code paths but make no call to `assert`, no `pytest.raises`, and no `unittest.mock.assert_*` method. They provide execution-smoke value only — they verify the code does not raise an unhandled exception, which is a weaker guarantee than any behavioral assertion. The top 30 are scheduled for deletion in the near term. The 332 `pytest.raises`-only tests are lower priority: they verify that a specific exception type is raised, which is meaningful for error-handling paths, but the absence of a message or state assertion means they cannot distinguish a correct error from an accidental one sharing the same type. The 20 `provider_name_only` tests assert only that a provider's `name` attribute equals a string constant; this is a configuration-identity assertion with near-zero regression value. Cleanup of the `provider_name_only` category is deferred to a follow-up pass.

### Written-once files

The written-once file audit is complete. No additional action items were surfaced beyond what is captured in the sections above.

## Recommended actions

| Category | Count | Action |
|---|---|---|
| Characterization tests — confirmed duplicates | 5 | Delete immediately; duplicate coverage identified per test |
| True zero-signal tests | 30 | Delete in current sprint |
| `provider_name_only` tests | 20 | Defer to follow-up; low cost to fix but not blocking |
| `pytest.raises`-only tests | 332 | Audit in follow-up; add message/state assertions where the error path is load-bearing |
| One-assert-no-mock tests | 1789 | No action; single-assertion tests are acceptable if the assertion is behavioral |

## Decision

This ADR commits to the following immediate actions: delete all five confirmed-duplicate characterization tests listed in the table above, and delete the top 30 true-zero-signal tests identified by the audit. The `provider_name_only` cleanup (20 tests) and the `pytest.raises`-only hardening pass (332 tests) are deferred to a follow-up task to avoid scope creep in the current sprint. No changes are made to one-assert-no-mock tests unless a specific test is found to be a duplicate or zero-signal by a stricter criterion.
