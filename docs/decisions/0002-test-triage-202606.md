# ADR 0002 — Test Triage 2026-06

## Status

Accepted, revised 2026-06-15 — supersedes the original 2026-06 version of this ADR in place. The prior text is preserved in git history.

## Context

The original ADR-0002 recorded a June 2026 triage audit of the Ratatoskr test suite (then 696 files, ~134K LOC) and committed to an immediate cleanup: delete five duplicate characterization tests and the top 30 true-zero-signal tests, while deferring `provider_name_only` and `pytest.raises`-only hardening.

That immediate cleanup has shipped. This revision is a **re-measurement by fact** as of 2026-06-15. The characterization deletions and the bulk of the low-value pruning are done; the headline numbers shrank accordingly. Re-running the audit produced two corrections to the original framing:

1. **The "30 true zero-signal" was a hand-picked top slice, not a population count.** A consistent AST pass over the current tree finds **81** functions with no assertion signal of any form (no `assert`, no `pytest.raises/warns`, no `unittest assert*`, no mock `.assert_*()`). The 30 → 81 move is **not** a regression — 81 is the full current population. Separately, **604** functions have no bare `assert` statement ("total zero-assert"), but most of those *do* check behavior via `pytest.raises` or a mock `.assert_*()`. So **zero-assert (604, broad) and true-zero-signal (81, strict subset) are distinct categories** — the original "684 total zero-assert" reproduces as 604 today (the suite shrank); it did **not** collapse into the 81.
2. **The original "delete the top 30 zero-signal" over-estimated deletable dead weight.** Stratified sampling (below) shows the 81 zero-signal hits overstate truly worthless tests by roughly **7×**: most are either real assertions the heuristic cannot see, or legitimate no-raise / no-op / error-swallow contracts.

This revision records the current measured state, the disposition of the original commitments, and a right-sized decision that prefers "delete the confirmed-worthless subset, defer the rest with rationale" over a large hardening sweep.

## Findings

### Metrics: BEFORE → NOW

Counts measured 2026-06-15 via a single AST pass over `tests/` (662 total `.py` files, 606 of them `test_*.py` / `*_test.py`). All NOW figures are reproducible by the method described at the end.

| Metric | BEFORE (orig. 2026-06) | NOW (2026-06-15) |
|---|---|---|
| Test files (`test_*.py` / `*_test.py`) | 696 | 606 |
| Test LOC (all `.py` under `tests/`) | ~134K | ~126K |
| Test functions | not measured | 4,857 |
| True zero-signal (no assertion of any form) | 30 | 81 |
| `pytest.raises`-only (only assert-signal is `raises`) | 332 | 282 |
| `provider_name_only` (sole assert on an identifier string) | 20 | 13 |
| Total zero-assert (no bare `assert`; may use raises/mock) | 684 | 604 |
| One-assert-no-mock | 1789 | 1,706 |
| Flaky markers (`flaky` / `rerunfailures`) | 0 | 0 |
| Characterization tests | 5 (in `tests/characterization/`) | 0 — directory **deleted** |

A finer detector (also recognizing `pytest.warns`, `assertRaises`, and `raises` paired with a mock `.assert_*()`) shifts a few counts by small amounts — e.g. pure-type-only `raises` ≈ 231 of the 282 raises-bearing tests, `provider_name_only` ≈ 14, one-assert ≈ 1,735. The decision below is insensitive to these deltas.

### Characterization tests — DONE

`tests/characterization/` no longer exists. All five originally-flagged duplicates are gone and **none have been reintroduced**: no `*_shape_is_stable` function, no golden-fixture function, no equivalent reappears anywhere in the tree. The `@pytest.mark.characterization` tests that remain under `tests/adapters/git_backup/` are deliberate ports of Kotlin reference tests for distinct git-backup behaviors — not ADR-0002 residue and not duplicates.

### Zero-signal tests — re-characterized

Stratified sampling of 25 zero-signal hits across all directories found a **strict-worthless rate of ~4% (1/25)** and a **redundant-near-duplicate-smoke rate of ~8% (2/25)**. The remaining ~80% are:

- **Real assertions the heuristic cannot see** — the dominant false positive. `pytest.fail()` inside `if failures:` (the `tests/api/test_openapi_*.py` drift/security suites, ~14 of 81), assertions delegated to a called helper (the entire `tests/api/test_media_snapshot.py` block via `_assert_media`, 7 of 81), and `try: …; pytest.fail("should have raised"); except: pass` frozen-dataclass tests (`tests/test_vector_store_contract.py`). These are strong contract tests, not dead weight.
- **Legitimate no-raise / no-op / idempotency / error-swallow contracts** — for an error-swallowing or no-op function, "did not raise" *is* the contract (e.g. `test_rss.py`, `*_cancellation.py`, `test_redis_lock.py`, Prometheus-absent no-ops in the observability tests, `test_db_write_queue.py` idempotency). These guard real regressions and should stay; they would be *hardened* (an explicit side-effect or `caplog` assertion), not deleted.

Concrete genuinely-worthless examples (the deletable subset):

1. `tests/test_scraper_providers.py::test_aclose_is_noop` — calls `await provider.aclose()` and asserts nothing; `PlaywrightProvider.aclose` is literally `pass`. A no-op asserting a no-op; can never fail.
2. `tests/test_response_size_validation.py::test_response_at_exact_limit` — duplicate of `test_valid_response_under_limit` (same "no raise on a small Content-Length"); adds no boundary the limit-exceeded test doesn't already pin.
3. `tests/test_response_size_validation.py::test_openrouter_response_validation` — same "under-limit → no raise" assertion with a different service-name string; pure redundancy. (Keep `test_valid_response_under_limit` as the canonical negative-branch smoke for that security guard.)

**Defensible deletion-worthy estimate among the 81: ~8–12 tests (~10–15%)** — roughly 3–5 strictly worthless (assert a no-op/empty impl) plus 5–8 redundant near-duplicate smoke. The other ~70 are real assertions the AST pass cannot see (~21) or legitimate no-raise/no-op contracts (~48). The "81 zero-signal" headline overstates deletable dead weight by ~7×.

### `pytest.raises`-only (282) — tiered

Of the 282 tests whose only assert-level signal is `pytest.raises`, the survey found ~60 actually pair it with a mock `.assert_*()`/state check (so they are not truly type-only), leaving ~220 pure type-only. Within the pure type-only set:

- **(A) Type-is-the-contract (leave):** code that maps different inputs to different exception *types*, where a wiring bug surfaces as the wrong type — e.g. `tests/api/test_proxy.py` (`ValidationError` vs `ResourceNotFoundError` vs `ExternalAPIError`), `tests/security/test_token_crypto.py` (`MissingEncryptionKeyError` vs `InvalidEncryptedTokenError`), HTTP-status→typed-error mappings in `test_github_api_client.py` / `test_reddit_ingester.py` / `test_tts_client.py`, and `test_mark_summary_as_unread.py`.
- **(B) Security/SSRF guards worth hardening (~6):** bare `ValueError` where a silent-pass regression is dangerous — chiefly `tests/test_git_url_safety.py` (`assert_safe_git_url` / `assert_resolved_public_host`, especially the private-rebind case). Add `match=` / an `excinfo.value` message assertion so a future refactor can't accept a private rebind by raising `ValueError` elsewhere.
- **(C) Pydantic boundary tail (~50):** per-field min/max/zero/negative `ValidationError` one-liners that re-test Pydantic's own `Field(ge=, le=)` constraints — `test_slow_handle_knobs.py`, `test_scoped_goals.py`, `test_bulk_operation_contracts.py`, `test_rss_config.py`, `test_repo_analysis_contract.py`, `test_qdrant_schemas.py`. Near-zero individual value but cheap; **consolidate into `@pytest.mark.parametrize`**, don't individually delete or harden.

### `provider_name_only` (13)

- **4 are misclassified behavior tests (leave):** priority-sort ordering in `tests/adapters/git_backup/test_priorities_ignore_repack.py` and `test_mirror_service_coverage.py`, idempotency-key reuse in `test_transcription_job_queue_postgres.py`, and the unpredictable-backup-filename security check in `test_system_maintenance_service.py`.
- **~9 are genuine config-identity** (`test_scraper_providers.py`, `tests/unit/scrapers/*`, `test_api_request_models.py`). Caveat: a scraper's `provider_name` string is reused as the `endpoint=` metrics label and the `crawl_results.endpoint` DB column, so a rename would silently re-label observability/DB rows. Keep **one anchor test per distinct provider string** as a contract pin; delete only the redundant second variant (e.g. drop `test_provider_name_default`, keep `test_provider_name_returns_configured_name`).

### Written-once files

130 substantive test files (single add-commit, never modified). The vast majority are healthy: high assertion-density suites whose targets are stable (the OTel observability trio landed 2026-05-30, the `git_backup/` suite 2026-05-29, the chaos circuit-breaker test, and dense pure-logic suites like `test_rule_actions.py`, `test_import_parsers.py`). Written-once is not itself a defect.

The exception is a **coverage-farming scaffolding cluster** created in three coverage-targeting commits (`test: raise coverage floor to 65 percent`, `test: raise package coverage above 50 percent`, `fix(ci): restore coverage and formatting gates`) and never touched since. The clearest dead scaffolding:

- `tests/infrastructure/persistence/repositories/test_repository_adapters_smoke.py` — `inspect`-driven reflection loop calling every coroutine adapter method against a fake empty DB, asserting only `attempted >= 80`.
- `tests/mcp/test_services_smoke.py` — same pattern, asserts `attempted >= 20`.
- `tests/db/test_sqlalchemy_dependency_smoke.py` — `assert select(1) is not None`; tests SQLAlchemy itself, not app code.

These move a coverage gate without verifying behavior and are deletion candidates.

### Duplicate clusters (pre-existing, not new)

Two provable "same envelope/shape, same function, different mock depth" clusters remain (the DB/HTTP-backed test in each subsumes the shallow re-tests):

- **Sync wire-key envelope:** `tests/api/test_sync_apply_contract.py` (snake-case key tests at `:188`/`:266` and the 304/ETag trio at `:348`/`:388`/`:408`) are shallow re-tests of the DB-backed equivalents in `tests/api/test_sync.py` (`:131`, `:625`, `:663`, `:701`). `test_sync_v2_contract.py` (the canonical contract file the original ADR redirected coverage to) does **not** overlap.
- **Request-lifecycle status projection:** `tests/api/test_request_response_lifecycle_contract.py` (`:14`/`:30`/`:48`) and the endpoint pair `tests/api/test_requests_status_endpoint.py:42` + `tests/api/test_content_router_authorization.py:73` re-assert the projection already owned by the parametrized mapper test `tests/api/test_request_lifecycle_mapping.py:33` and the DB test `tests/api/test_request_status.py:141`.

## Decision (2026-06-15)

The original ADR's immediate actions are complete. This revision adopts a **delete-the-confirmed-worthless, defer-the-rest-with-rationale** posture and explicitly avoids a large hardening sweep.

1. **Zero-signal (81):** delete only the confirmed-worthless subset (~8–12 tests) — the no-op-asserting-no-op tests (`test_aclose_is_noop`) and redundant near-duplicate smoke (the two `test_response_size_validation.py` duplicates, keeping the canonical one). Do **not** delete the ~48 legitimate no-raise/no-op/error-swallow contracts; harden them opportunistically (explicit side-effect or `caplog` assertion) only when the surrounding file is already being edited. Take no action on the ~21 false positives (`pytest.fail`/helper-delegated/`try-fail-except`) — they are correct tests the heuristic cannot see.
2. **`pytest.raises`-only (282):** leave tier (A) type-is-the-contract tests and the ~60 already-paired tests as-is. Harden the **~6 security/SSRF guards** in tier (B) with `match=`/`excinfo.value` assertions (chiefly `test_git_url_safety.py`). **Consolidate** the ~50 Pydantic boundary one-liners in tier (C) via `@pytest.mark.parametrize` rather than deleting or hardening them. No outright deletions.
3. **`provider_name_only` (13):** keep the 4 behavior tests and one anchor per distinct provider-name string (the strings double as observability/DB labels); delete only redundant second variants.
4. **Written-once scaffolding:** delete the three reflection/dependency smoke files (`test_repository_adapters_smoke.py`, `test_services_smoke.py`, `test_sqlalchemy_dependency_smoke.py`) and the thinnest coverage-push files, since they move a coverage gate without asserting behavior. Leave all high-density single-commit suites untouched.
5. **Duplicate clusters:** drop the shallow `_without_db`/model-dump re-tests in the two clusters above (the `test_sync_apply_contract.py` snake-case + 304 trio; `test_request_response_lifecycle_contract.py`); keep the DB/HTTP-backed and canonical-mapper tests.
6. **One-assert-no-mock (1,706):** no action — a single behavioral assertion is acceptable.

Net committed deletions are small and confirmed (~15–25 tests across categories); the larger boundary tail is consolidated, and load-bearing contracts are kept. This deliberately under-commits on hardening to avoid scope creep, with each deferral justified above.

## Method

All counts are reproducible via a single AST pass over `tests/`: walk every `.py`, collect functions whose name starts with `test`, and per-function tally `ast.Assert` statements, `with pytest.raises/warns/deprecated_call`, `unittest assert*`/`assertRaises` calls, and mock `.assert_*()` calls. *True zero-signal* = none of those present (81). *Total zero-assert* = no `ast.Assert` statement, regardless of `raises`/mock (604) — a strict superset of zero-signal. Known blind spots (and the source of the false positives above): the pass does not follow calls into helper functions, does not recognize `pytest.fail()` as an assertion, and does not model fixtures. No analysis tooling was committed; the pass is ephemeral.
