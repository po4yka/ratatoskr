## CRITICAL

### asyncio.run() called inside running event loop in export_summary handler

- **File:** `app/api/routers/content/summaries.py:779`  ·  _api-content_  ·  confidence 0.99
- **Problem:** The async FastAPI handler `export_summary` calls `formatter.export_summary()` (line 780), a synchronous method that internally calls `asyncio.run()` three times (export_formatter.py:84, 87, 89) to execute async DB queries. Calling `asyncio.run()` inside an already-running event loop raises `RuntimeError: This event loop is already running`, which will always crash the handler at runtime. This is a hard failure path for every export request.
- **Fix:** Convert `ExportFormatter.export_summary` and its `_load_summary` helper to `async def` and `await` the repository calls directly. Update the handler to `await formatter.export_summary(...)`. Remove all `asyncio.run()` calls from the formatter.

## HIGH

### Blocking synchronous HTTP in async endpoint: PyJWKClient fetches Apple JWKS on every request

- **File:** `app/api/routers/auth/apple.py:140`  ·  _auth-security_  ·  confidence 0.98
- **Problem:** jwt.PyJWKClient(_APPLE_JWKS_URL).get_signing_key_from_jwt(token) performs a synchronous HTTPS request to https://appleid.apple.com/auth/keys inside an async FastAPI route handler. This blocks the entire asyncio event loop for the duration of the DNS lookup + TLS handshake + HTTP round-trip. Under any load this will stall all other concurrent requests. Additionally, a new PyJWKClient is instantiated on every callback with no caching, so every Apple login hits Apple's JWKS endpoint.
- **Fix:** Wrap in asyncio.to_thread: `signing_key = await asyncio.to_thread(jwt.PyJWKClient(_APPLE_JWKS_URL).get_signing_key_from_jwt, token)`. Better: cache a module-level PyJWKClient with cache_keys=True (PyJWT supports this) and wrap in asyncio.to_thread, or use httpx.AsyncClient to fetch and parse the JWKS asynchronously.

### run_decoy_verify passes raw (non-peppered) password to hasher.verify, breaking timing parity

- **File:** `app/api/routers/auth/credential_auth.py:260`  ·  _auth-security_  ·  confidence 0.97
- **Problem:** run_decoy_verify calls _get_hasher().verify(_get_decoy_phc(), password) where `password` is the raw plaintext from the request. But the real verify path in verify_password first runs _pre_hash(password, pepper) and then calls hasher.verify(phc, digest) against the HMAC-SHA256 hex digest. The decoy PHC was hashed from the literal string 'decoy-not-a-real-password' (line 86), which has no pepper. The argon2 library hashes its input, so verifying a 64-char hex digest vs verifying a variable-length raw password produces different argon2 execution times (different salt derivation path and input length), weakening the timing-parity guarantee that is the whole point of the decoy. An attacker can distinguish 'identifier not found' from 'wrong password' via timing.
- **Fix:** In run_decoy_verify, apply the same pre-hash before verify: `digest = _pre_hash(password, _get_credentials_pepper()); _get_hasher().verify(_get_decoy_phc(), digest)`. Also generate the decoy PHC from a pre-hashed string so input length matches: `_decoy_phc_holder[0] = _get_hasher().hash(_pre_hash('decoy-not-a-real-password', _get_credentials_pepper()))`.

### PlaywrightProvider runs sync Playwright in asyncio.to_thread but uses no per-call semaphore, enabling unbounded concurrent browser launches

- **File:** `app/adapters/content/scraper/playwright_provider.py:153`  ·  _scraper-content_  ·  confidence 0.97
- **Problem:** Each call to `PlaywrightProvider.scrape_markdown` dispatches `_render_html_sync` via `asyncio.to_thread`, which launches a full Chromium browser process. ### async_increment_failure_count: non-atomic read-modify-write under concurrency

- **File:** `app/infrastructure/persistence/repositories/webhook_repository.py:152`  ·  _persistence_  ·  confidence 0.9
- **Problem:** async_increment_failure_count reads the current failure_count with a scalar SELECT, increments in Python, then writes back with a separate UPDATE. Two concurrent delivery failures for the same subscription will both read the same count value, both compute count+1, and both write the same result — effectively losing one increment. Under any load that causes concurrent delivery attempts this leads to a failure_count that is systematically lower than the true failure count, potentially preventing automatic subscription disablement.
- **Fix:** Replace the read-compute-write pattern with a single atomic SQL expression: `update(WebhookSubscription).where(...).values(failure_count=WebhookSubscription.failure_count + 1, last_failure_at=_utcnow())`. Remove the SELECT. This eliminates the race without needing any locking.

### Race condition: asyncio.Lock created outside the lock in ensure_api_runtime

- **File:** `app/mcp/context.py:204`  ·  _mcp_  ·  confidence 0.95
